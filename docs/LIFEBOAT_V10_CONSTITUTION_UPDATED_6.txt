On February 16, 2026, Kane Mayfield's MetaMask wallet (kanemayfield.eth)
was compromised. The seed phrase was exposed. A sweeper bot was deployed —
monitoring the wallet address 24/7 and instantly draining any ETH the
moment it arrived. The bot had no interest in NFTs. NFTs require gas to
move. No ETH = no gas = NFTs trapped.

Over approximately 18 hours — in real time, under real pressure, with
real assets at stake — Kane and Claude built two tools from scratch,
debugged them through live failures, and executed a successful rescue
operation:

│ • 115 ETH mainnet NFTs confirmed                                        │
│ • 261 Polygon NFTs confirmed                                            │

│ • Manifold contract ownership transferred                               │

│ • Total: 413 NFTs rescued                                               │
│ • Compromised wallet drained to zero — bot has nothing left to sweep    │
└─────────────────────────────────────────────────────────────────────────┘

The tools that made this happen were originally called RESCUE (the scanner)
and LIFEBOAT (the executor). V10 merges both into a single unified
application with a local web server architecture. The scanner and executor
are now tabs in one interface — same proven engine, better experience.

Nobody should have to go through this without a tool designed for exactly
this scenario. That is the mission.

── V10 ARCHITECTURE — LOCAL WEB SERVER ────────────────────────────────────

V10 is a single unified application. The user downloads a folder,
double-clicks a launcher, and their browser opens to localhost:3000.
One window, one experience. All blockchain operations happen through a
local Node.js server — private keys never leave the user's machine,
and MEV Blocker works without CORS issues.

┌─────────────────────────────────────────────────────────────────────────┐
│ ARCHITECTURE OVERVIEW                                                   │
│                                                                         │
│ User runs start.bat (Windows) or start.sh (Mac/Linux)                   │
│   → Node.js Express server starts on localhost:3000                     │
│   → Browser opens automatically                                        │
│   → V8 visual interface loads (single HTML file)                        │
│   → All blockchain calls route through server.js → engine.js            │
│   → MEV Blocker hides ETH transactions from sweeper bot                 │
│   → Keys exist only in memory, never written to disk                    │

── RESCUE TAB — The Scanner ───────────────────────────────────────────────

A tab within the unified V10 interface. Scans compromised wallets for
NFTs across all supported chains. No separate file — integrated into
the single-page application.

Core Scanning:
  • Multi-chain sequential scan: Ethereum, ApeChain, Base, Arbitrum,
    Optimism, Avalanche, Polygon — one chain at a time via Alchemy API.
    SSE progress stream shows each chain lighting up in real time.
    Scan order is fixed: ETH → ApeChain → Base → Arbitrum → Optimism →
    Avalanche → Polygon (Polygon last because it has the most spam).
  • Alchemy API key field: User provides their own key. Graceful error
    if missing or invalid. FAQ and README include step-by-step for
    enabling all chains (Alchemy defaults to ETH-only on new apps).
  • FROM wallet field: The compromised wallet address to scan
  • TO wallet field: The clean destination wallet address
  • SCAN-ONLY MODE: Only the FROM wallet is required to scan. The TO
    wallet is gated at export and "Launch Lifeboat" time. This lets
    someone assess a compromised wallet without committing to anything —
    no keys, no destination needed just to look.
  • Address validation: Fields validated before their respective actions.
    Header shows both addresses once entered.
  • Global wallet inputs: FROM and TO are entered once at the top and
    shared across all tabs. Less repetition, less chance of typos.

NFT Display:
  • Gallery view + List view toggle: Switch between visual grid and
    compact list
  • Lazy-load images: Warning/spam badge fires only on onerror — NOT on
    slow IPFS load. This distinction matters. Slow load is normal; failed
    load is suspicious.
  • ERC-1155 quantity badge: Shows "x3" etc. for tokens held in quantity
  • OpenSea link per card: Quick verification without leaving the tool
  • Collection grouping: NFTs shown by collection within each chain
    section, with collapsible accordions
  • Chain sections: NFTs separated by chain (Ethereum, Polygon, Base,
    etc.) with colored headers showing chain name, badge, and tally
  • One-click collection select/deselect: Click the checkbox on a
    collection header to select/deselect ALL NFTs in that collection
    at once. Shows full/partial/empty selection states.
  • Collapse/expand collections: Click the collection header to
    show/hide the NFT grid. Prevents large collections from
    monopolizing the screen.

Selection & Filtering:
  • Check/uncheck per NFT: User controls exactly what gets included
    in the rescue JSON
  • Spam/flag button per card: Orange border state. Marks token as
    suspicious without removing it.
  • Filter tabs: All / Selected / Skipped / Flagged — lets user work
    through a large collection systematically
  • Search field: Filter by collection name or token name
  • Select All / Deselect All buttons: Bulk selection controls
  • Header stats: Found / Rescue / Skip counters at top

Export & Import:
  • Export JSON: Produces rescue_data.json with rescue_from, rescue_to,
    chain keys, collection data, token IDs, ERC-1155 balances.
  • Export CSV: For record-keeping or cross-reference
  • Import previous session JSON: Resume an interrupted scan. Critical
    for large wallets — scanning 400+ NFTs takes time.
  • Session auto-save to localStorage: If browser crashes mid-scan,
    don't lose 10 minutes of work. Cleared on burn.


── LIFEBOAT TABS — The Executor ───────────────────────────────────────────

The execution side of V10. Multiple sub-tabs for different rescue
operations, all powered by engine.js through the local Express server.

NFT Rescue Tab:
  The primary execution flow. Reads rescue data from the RESCUE tab
  (or imported JSON), estimates gas, funds the compromised wallet,
  executes transfers, and waits for confirmations.

  • Private key entry: User enters compromised key and funding key.
    Keys never leave localhost. Placeholder text catches unfilled
    keys with a clear error.
  • Per-token estimateGas with buffer: CRITICAL. Every token gets its
    own gas estimate via eth_estimateGas before the transaction is
    built. This is what makes non-standard contracts (BlazedCats,
    Kindred Hearts, Manifold custom deployments) work. Flat gas limits
    cause out-of-gas failures on ~20% of contracts.
  • Tokens that would revert are skipped cleanly: estimateGas failure =
    token already moved, burned, or restricted. Logged as warning,
    not counted as failure. No wasted gas.
  • Balance check button: Before any chain execution, shows live
    balances for BOTH wallets. FROM (compromised — red) and TO
    (clean — green). Color-coded: red if funding wallet is below
    required gas, green if covered.
  • Explorer links in execution log: Each submitted transaction gets
    a clickable Etherscan/Polygonscan link so user can verify on-chain.
  • Pre-execute summary modal: "You are about to move X NFTs from
    0x123... to 0x456... This will cost ~0.03 ETH." One last
    sanity check.
  • Post-rescue checklist: "🚢 Your lifeboats have reached safety."
    Then: Delete your keys from this tool NOW. Never use compromised
    wallet again. Consider hardware wallet.

Gas Tier Selector:

│                                                                         │
│ Users cannot set gas lower than Recommended. People who just lost       │
│ money may "frugal too hard" and cause failed transactions. We help      │
│ people, but we don't open a window for Harry Haggle to negotiate        │
│ himself into failure.                                                   │
│                                                                         │
│ Recommended (1.1x) — Default. The floor. What proved itself.            │
│ Safe (1.25x)       — For busy networks or non-standard contracts.       │
│ Maximum (1.5x)     — When you absolutely cannot afford a failure.       │
│                                                                         │

Token Sweep Tab:
  Separate from NFT rescue. Scans ALL chains simultaneously for ERC-20
  tokens (USDC, WETH, random airdrops, etc.) using Alchemy's
  alchemy_getTokenBalances API. Shows token name, symbol, balance, and
  chain for each found token. Per-token Sweep button transfers the
  full balance to the destination address. Engine functions:
  scanAllTokens (scanning) and sweepToken (execution).

Native Balance Dashboard (formerly Native Sweep Tab):
  Scans ALL chains at once for native token balances (ETH, POL, AVAX,
  etc.) and displays a dashboard with one row per chain showing: chain
  name, balance, symbol, MEV shield indicator (ETH only), and a
  per-chain Sweep button. Replaces the old single-chain dropdown
  approach. Hit scan once, see everything, sweep what's worth sweeping.
  Engine function: getAllBalances (scanning), sweepETH (execution).

Manifold Tab:
  Handles Manifold creator contract admin transfer. Funds compromised
  wallet via MEV Blocker, immediately fires transferOwnership(newOwner).
  IMPORTANT: Only transferOwnership goes through MEV Blocker from the
  compromised wallet. approveAdmin and setRoyalties must be called from
  the clean wallet through Manifold's own UI after ownership transfers —
  the old wallet loses admin rights the moment ownership moves.

Quiet Fund Tab:
  Sends a precise amount of native gas to the compromised wallet through
  MEV Blocker's private relay on Ethereum mainnet, or standard RPC on
  other chains. This is the prerequisite for any manual operation that
  requires the compromised wallet to sign — ENS transfers, DeFi unstaking,
  marketplace delisting. The tab includes a full "READ THIS FIRST" warning
  explaining the fund-and-fire choreography: have the third-party site
  loaded and ready to sign BEFORE sending gas. On Ethereum, the bot
  typically has seconds to react. The tab explains this without softening it.

Other Assets Tab:
  A content-rich guide for rescue operations that don't fit the standard
  NFT transfer flow. Three sections:

  ENS Names (.eth): Explains why safeTransferFrom fails (registrar rejects
  it), what the three components are (Owner, Manager, ETH Address), and
  the exact fund-and-fire method used to rescue kanemayfield.eth. Includes
  an inline button that jumps to the Quiet Fund tab mid-walkthrough.

  Exchange-Locked & Staked Tokens: DeFi stakes (Lido, Aave, Compound)
  require unstaking from the compromised wallet first — same Quiet Fund
  choreography. Cooldown periods documented. Centralized exchanges
  (Coinbase, Binance) covered separately — if email/2FA is intact,
  withdraw directly, no LIFEBOAT needed.

  Other Domain Names: Unstoppable Domains (.crypto, .nft, .x) on Polygon
  and .base names on Base are standard NFTs — normal scan finds them.
  .sol names are Solana — not supported.

Emblem Vault Tab:
  Three sub-sections for Emblem Vault NFTs on Ethereum mainnet:

  Scan & Rescue: Scans both V2 contract (0x82C7a8f7...) and Legacy
  contract (0x6Fc355D4...) for vaults owned by the compromised wallet.
  Shows vault name, image, and contained assets (BTC, DOGE, rare pepes,
  etc.) before execution. Transfers via MEV Blocker — same pattern as
  NFT Rescue tab.

  Vault Inspector: Read-only lookup by token ID. No keys needed. Shows
  name, image, contained assets, full metadata.

  Ownership Proof: Generates a cryptographic EIP-712 signature proving
  vault ownership. Zero gas, completely invisible to sweeper bot. Used
  for profile migration with Emblem support without revealing private key.

Fractal Visions Tab — Rescue Suite:
  A dedicated two-section tab for the Fractal Visions NFT ecosystem.
  Exists separately because Fractal Visions operates on Superchain networks
  that Alchemy does not index. Different data source, same rescue engine.

  Collector Rescue (Section 1): Scans Soneium, Shape, Superseed, and
  Unichain simultaneously via each chain's Blockscout REST API
  (/api/v2/addresses/{wallet}/nft). No API key required. Returns NFTs
  with images, collection names, token IDs, and balances. Executes rescue
  using the standard /api/execute engine — same transfer logic, different
  chains. WARNING: these chains have no MEV protection. Transfers are
  visible in the mempool. Document this clearly in the UI.

  Creator Rescue (Section 2): Queries the Fractal Visions Launchpad
  registry (0x7A10b7d6Dc513Ae5F98f2C1546269Be9DE94ebD3 — same address
  on all 7 FV chains, deterministic deployment) for ERC721 and ERC1155
  collections owned by the compromised wallet. Calls transferOwnership()
  directly on each collection contract. Separate ABIs required for each
  token type — see Known Failures.

── MARK V: ARCHITECTURE DECISIONS ────────────────────────────────────────

  MEV Blocker: NOT used in Mark V.
    Clean wallets have no sweeper bot. Standard RPC for all chains.
    No MEV overhead, no latency penalty, no bundle complexity.

  Alchemy Pro required:
    Mark V fires simultaneous scans across up to 50 wallets x 7 chains.
    That is potentially 350 concurrent Alchemy API calls. A free Alchemy
    key rate-limits instantly. This is stated clearly in the UI BEFORE
    the user enters anything. Link to Alchemy pricing included.
    Minimum: Alchemy Growth plan.

  Parallel execution across wallets:
    Each wallet has an independent nonce counter. Wallet 1 and Wallet 7
    can execute simultaneously — the blockchain has no constraint here.
    Within each wallet, transfers are sequential (nonce-ordered).
    Think of it as: N lanes of traffic, each lane moving in order.

  Queue manager for API backpressure:
    Global concurrent call limit (~20 simultaneous) prevents Alchemy
    from returning 429 errors. Requests queue and drain — not dropped.

  Nibble approach for large fleets:
    Gas estimation and execution happen in batches per wallet, not all
    at once. Prevents timeouts and partial failures on very large
    operations (50 wallets x 500 NFTs each).

── MARK V: EMBLEM VAULT OPERATIONS ────────────────────────────────────────

  This section documents the full unvault flow as researched and confirmed
  with Emblem's AI in May 2026. Every step below is verified.

  SECTION 1 — BULK VAULT TRANSFER (EVM ONLY):
    Scans multiple wallets for Emblem Vault NFTs (V2 + Legacy contracts).
    Aggregates all vaults into one fleet view.
    Transfers vault NFTs to a destination wallet in bulk.
    Uses fetchVaultsOfType from emblem-vault-sdk for scanning —
    cleaner than Alchemy NFT API for vault-specific data.
    Execution uses standard engine.js transfer logic (same as NFT Rescue).
    No MEV Blocker needed (clean wallets).
    No new unvault dependencies required for this section.

  SECTION 2 — UNVAULT AND XCP ASSET EXTRACTION:
    Allows the operator to unvault Emblem Vaults and move the contained
    XCP/Bitcoin assets to a native Counterparty wallet.

    THE FULL CONFIRMED UNVAULT FLOW (verified from SDK source, May 2026):

    Step 1 — Personal Sign (local, ethers.js):
      wallet.signMessage("Claim: {serialNumber}")
      Pure local. No gas. No transaction. No network call.
      serialNumber comes from vault metadata.

    Step 2 — Signer Proxy (HTTP POST, Node.js fetch):
      POST https://tor-us-signer-coval.vercel.app/sign
      Headers: { "Content-Type": "application/json", "chainid": "1" }
      Body: { signature, tokenId }
      Returns: { token: "jwt..." }
      CONFIRMED OPEN: Live test May 18, 2026 returned HTTP 400 (bad
      signature format as expected). No auth headers required. No API
      key. No CORS restriction. Plain Node.js fetch confirmed working.

    Step 3 — Torus Key Retrieval (npm packages):
      getTorusKeys(tokenId, jwt.token)
      Uses: @toruslabs/fetch-node-details + @toruslabs/torus.js
      TORUS_VERIFIER  = 'tor-us-signer-vercel'
      TORUS_CLIENT_ID = 'BOqGGv-Yx7Dp5RdvD9R3DgSC8jv66gQGwT3w22L7fj3Fg5WQ8AEUjJzyyEwD-qvq5eUQiVipyzOmRZTWBAxaoj0'
      Both values hardcoded in public SDK source. No secrets. No request.
      Returns: { privateKey: { privKey: "0x..." } }
      Calls Torus node network directly. No browser. No OAuth. No popup.

    Step 4 — AES Decrypt (local, CryptoJS):
      fetchMetadata(tokenId) -> metadata.ciphertextV2
      CryptoJS.AES.decrypt(ciphertextV2, privKey)
      Returns: { phrase: "12 word mnemonic" }
      Pure local. No network call.

    Step 5 — BIP44 Derivation (local, bip39 + hdkey):
      Derive Bitcoin/XCP private key from mnemonic.
      Derivation path: m/44'/0'/0'/0/0 (Bitcoin standard, same for XCP)
      SDK utilities: phrasePathToKey, generateTaprootAddressFromMnemonic

    Step 6 — XCP Asset Movement (xchain.io API):
      Construct Counterparty 'send' transaction with OP_RETURN data.
      Broadcast as raw Bitcoin transaction via xchain.io free API.
      BTC fees: fund the internal Bitcoin wallet first — same pattern
      as gas funding in Rescue Lifeboat, just on Bitcoin.

    REQUIRED npm PACKAGES for Section 2:
      emblem-vault-sdk
      @toruslabs/fetch-node-details
      @toruslabs/torus.js
      crypto-js
      bip39
      hdkey

    IMPORTANT — Torus cannot be bypassed, but needs no browser:
      The Torus step is required. It does NOT need OAuth or Google login.
      The signer proxy (Step 2) handles ownership verification via EVM
      signature. Everything runs in Node.js. Local-only architecture
      is fully confirmed viable.

    WHAT COMES OUT:
      A BIP39 mnemonic phrase. Standard. Importable into any Bitcoin
      wallet. From this, all XCP assets in the vault are accessible.


── MEV Protection ─────────────────────────────────────────────────────────

  • ETH Mainnet: MEV Blocker direct RPC (rpc.mevblocker.io):
    Transactions are submitted privately. The sweeper bot cannot see
    funding or transfers in the mempool. No bundle size limit. This is
    the proven method — the one that actually worked.

  • All other chains: Standard RPC: ApeChain, Base, Arbitrum, Polygon,
    Optimism, Avalanche do not have the same sweeper bot risk. Standard
    RPC submission is acceptable and proven.


── Rate Limiting ──────────────────────────────────────────────────────────

  Scan rate limiting (alchemyFetch layer):
  • alchemyFetch(): armored HTTP wrapper for ALL Alchemy API calls. Retry
    3x with exponential backoff (1s, 2s, 4s) on 429/5xx. 30-second timeout.
    Structured errors. No direct fetch() to Alchemy anywhere else in
    engine.js — every call routes through this one function.
  • paginatedNFTFetch(): wraps alchemyFetch for getNFTsForOwner. Follows
    pageKey until done. 200ms pause between pages. 50-page safety cap.
    Per-page progress callback for SSE reporting.
  • Sequential chain scanning prevents rate limit collisions — only one
    chain hits Alchemy at a time.

  Execution rate limiting (unchanged):
  • Pause every 5 transactions: Prevents Alchemy 429 rate limit errors
    on large wallets (261 Polygon NFTs requires this)
  • Extended pause on 429 error: 3-second additional wait before retry
  • 2-minute confirmation timeout per tx: Prevents hanging indefinitely.
    Polygon and Base confirmations can be slow.

── GAS LOGIC — EXACTLY HOW IT WORKS ──────────────────────────────────────

During estimation (planning mode):

  1. Attempt eth_estimateGas per token using actual FROM/TO addresses
  2. If estimateGas succeeds: use result × gas tier buffer
     (1.1x / 1.25x / 1.5x)
  3. If estimateGas fails (wallet empty, token restricted): fall back
     to flat values (150k ERC-721, 200k ERC-1155) so estimate still
     produces a useful planning number
  4. Sum all estimated gas, multiply by current gasPrice, report
     native token needed

During execution (real mode):

  1. Fund the compromised wallet from the clean wallet (Step 1)
  2. Per token: call estimateGas again — now the wallet has funds
     so estimates are accurate
  3. If estimateGas succeeds: build transaction with that gas limit
     × tier buffer
  4. If estimateGas fails: skip token, log warning — it would revert
  5. Sign and submit via the chain's configured RPC
  6. Wait for confirmation with 2-minute timeout per tx

── DECISION TREE — PRE-ANSWERED ───────────────────────────────────────────

When you hit a choice mid-build, check here before spinning out.

  Q: Should we use Flashbots SDK?
  A: No. MEV Blocker direct RPC at rpc.mevblocker.io. See Known Failures.

  Q: Should LIFEBOAT run in the browser?
  A: No. CORS prevents MEV Blocker calls from browser contexts. The local
     web server solves this — browser handles UI, Node.js handles blockchain.

  Q: Should we use flat gas limits?
  A: No. Per-token estimateGas is required. See Gas Logic section.

  Q: Should we attempt to move ENS via safeTransferFrom?
  A: No. ENS registrar rejects it. Detect, skip, log. Direct user to
     app.ens.domains.

  Q: ETH version: ethers v5 or v6?
  A: v6. Always. import { ethers } from "ethers". Package: ethers@6.

  Q: Polygon priority fee: how much?
  A: Minimum 30 gwei. Polygon rejects < 25 gwei tip.

  Q: What if estimateGas returns zero or fails during estimation?
  A: Wallet is empty. Fall back to flat values (150k/200k) for the
     estimate only. Don't skip — it may work once funded.

  Q: Keys swapped (wrong wallet showing as funder)?
  A: Common during setup. compromisedKey = 0x9a28...9E53 (empty/hacked).
     fundingKey = 0x5624...94D6 (clean one with money). UI verifies both.

  Q: Can users set gas below recommended?
  A: No. Floor is 1.1x. This is not a democracy.

  Q: Should we add Solana support?
  A: Not in V10. Completely different architecture. Future consideration.

  Q: Can Mark V be used with compromised wallets?
  A: No. Never. Gas sent to a compromised wallet in a bulk queue sits
     exposed while other wallets process. Bot will take it. Run standard
     Rescue Lifeboat first, then Mark V for cleanup.

  Q: Does Mark V need MEV Blocker?
  A: No. Mark V is clean wallets only. No sweeper bot risk. Standard RPC.

  Q: Can the Torus step in the Emblem unvault flow be bypassed?
  A: No. But it requires no browser, no OAuth, no Google login. The
     signer proxy (tor-us-signer-coval.vercel.app/sign) accepts plain
     Node.js POST requests. Confirmed via live test May 18, 2026.

  Q: What Alchemy plan does Mark V require?
  A: Growth plan minimum. Free keys rate-limit instantly at fleet scale.
     State this clearly in the UI before the user enters any data.

  Q: What is the Mark V execution model — simultaneous or sequential?
  A: Parallel across wallets (independent nonce counters — no blockchain
     constraint). Sequential within each wallet (nonce-ordered). Queue
     manager caps concurrent API calls at ~20 to prevent 429 errors.

  Q: Should long-running Mark V operations use regular POST or SSE?
  A: SSE. A 50-wallet fleet execute can take 10-20 minutes. A regular
     POST holds the connection open silently — OS can close idle sockets
     during confirmation waits. SSE streams progress live and uses a
     15-second heartbeat to keep the connection alive. Two-step pattern:
     POST payload → get jobId, GET /api/...-sse?jobId=xxx → stream.

  Q: Where does the Emblem unvault logic live?
  A: In markv-engine.js (executeEmblemUnvault). markv-emblem.js was
     planned but never built as a separate file. Do not create it.

  Q: Can npm audit fix --force ever be run safely?
  A: No. It silently downgrades ethers from v6 to v5, breaking
     ethers.getAddress() and every scan. It also overwrites pinned Torus
     versions. Never run npm audit fix --force on this project.

  Q: What happens if an import statement is not at the top of an ESM file?
  A: The module fails to load entirely. Node.js ESM requires all import
     statements at the top of the file before any other code. A mid-file
     import causes a silent cascade failure — the importing module fails,
     which fails its importer, which can corrupt the entire server startup
     without a clear error message. Always put imports at the top.

  Q: Dark/light mode?
  A: No. Keep it dark. Emergency tool, not a settings playground.

  Q: Should we use a single ABI for ERC721 and ERC1155 totalSupply?
  A: No. Different selectors. Use FRACTAL_ERC721_ABI and FRACTAL_ERC1155_ABI
     separately, selected by token type. See Known Failures.

  Q: Should staticNetwork: true only apply to MEV Blocker?
  A: No. Apply it to ALL providers. UUPS proxy contracts on OP-stack chains
     misbehave without it. getProvider() uses staticNetwork globally.

  Q: How do we scan Soneium, Shape, Superseed, Unichain for NFTs?
  A: Blockscout REST API. Not Alchemy. GET /api/v2/addresses/{wallet}/nft
     Free, no API key, same response structure on all four chains.

  Q: Should engine.js have console.log statements?
  A: No. Zero. Server.js does the logging. Engine internals are silent.
     Add logs to debug, remove before committing.
